基于Wireshark的HTTPS协议分析

发表于 2017-05-17

简介

HTTPS (Hyper Text Transfer Protocol over Secure Socket Layer)是在TCP协议与HTTP协议之间加入了安全层(SSL/TLS)，该层协议会在传输数据包之前与服务器进行协商加密算法，在数据传输的过程中对数据包进行加密。

HTTPS协议栈图

加密算法

根据秘钥类型的不同分为：对称加密与非对称加密

  • 对称加密
    • 对称加密是加密与解密均采用同一个秘钥，而且通信双方必须都获得该秘钥；
    • 对称加密算法一般是对明文进行固定长度分组，然后使用指定秘钥加密，从而组成密文；
    • 对称加密算法一般是公开的，秘钥的安全性需要由用户自己来保证；
    • 常用算法有：DES,AES等；
    • 对称加密的优点是加解密的速度快；
  • 分对称加密
    • 非对称加密是加密与解密采用不同的秘钥，被称为公钥与私钥，必须要而且配合使用；
    • 数据发送者使用公钥进行加密，数据接受者只有使用对应的私钥才能进行解密；
    • 常用算法有：ECC,RSA,DSA等
  • 不可逆加密
    • 顾名思义该算法只能单向加密，加密过程不需要使用秘钥，直接由加密算法生成密文；
    • 加密后的密文无法还原成明文，只能再次用相同的加密算法对明文进行加密，然后比较两次加密后的密文是否一致；
    • 不可逆加密不存在秘钥的传输与保存问题，适合在网络系统使用；
    • 常用算法有：MD5等；

TLS/SSL

  • TLS与SSL的关系
    • SSL(Secure Sockets Layer)是网景公司设计的主要用于Web的安全传输协议；
    • IETF将SSL协议进行了标准化，并将其称为TLS(Transport Layer Security)；
    • 之所以将而且并列称呼为TLS/SSL，就是因为二者是同一个目的协议的不同发展阶段；
  • TLS/SSL协议的基本设计思路是公钥加密法，即：客户端先向服务器端请求公钥，然后用公钥加密明文信息，服务器收到密文后，用公钥所对应的私钥来解密；
  • TLS/SSL协议的最终目的是在服务器端与客户端最终生成相同的会话秘钥，之后通信的数据都需要用该会话秘钥来进行对称加解密;
  • TLS/SSL协议通信双方是通过握手协商并传递3个随机数使用非对称密钥交换算法来产生会话密钥，请参考下图(RSA 握手过程)；

  • TLS/SSL协议栈分为以下几种协议：
    • TLS记录协议(TLS Record Protocol)，该协议为应用层提供数据封装、压缩、加密等功能支持；
    • TLS握手协议(TLS Handshark Protocol)，握手协议建立在记录协议之上，用于通信双方实际传输数据之前，进行身份认证、协商加密算法、交换密钥；
    • Change Cipher Spec协议，用来通知对端从handshake协议切换到record协议；
    • Alert协议，发送应答码；
    • Application Data协议，把应用层数据(如：Http、Smtp等）传入record协议层作处理并且传输；
  • TLS协议采用主从式架构模型，目的在于提供两个应用进程间，通过网络的一个不安全通道，来创建安全的连接，来交换数据；
  • TLS协议与应用层协议独立无关，高层应用协议(如：HTTP,FTP等)能透明的创建于TLS协议之上；
  • TLS协议在应用层协议通信之前就已经完成加密算法，通信密钥的协商与传输以及服务器认证等工作，在此之后的应用层协议传输的数据，都将被加密；
  • TLS协议实现方式：
    • 服务器与客户端使用统一的TLS协议通信端口(如：HTTPS的443端口)；
    • 客户端请求服务器连接到TLS时使用特定的协议(如：邮件，新闻等协议);

  • 客户端与服务器都同意使用TLS协议，他们之间通过使用一个握手过程来协商出一个有状态的连接用以传输数据，具体过程如下：

    1. 客户端连接到服务器要求建立安全连接，并且列出客户端支持的密码组合(加密算法和加密哈希函数)，握手开始；
    2. 服务器从密码组合列表中选择加密算法和哈希函数，并通知客户端；
    3. 服务器发送数字证书给客户端，该证书包含服务器名称,受信任的证书颁发机构(CA)和服务器的公钥；
    4. 客户端确认服务器颁发证书的有效性；
    5. 为了生成回话秘钥用于安全连接，客户端使用服务器的公钥加密随机生成的秘钥，并将其发送到服务器端，服务器使用自己的私钥解密随机回话秘钥；
    6. 利用随机数，服务器与客户端生成用于加密和解密的对称秘钥；
    7. 以上步骤就是TLS协议的握手，握手完毕后的连接是安全的，知道被关闭，任何一个步骤失败，TLS握手过程就会失败，并且断开所有连接；

  • 总结TLS包含三个阶段：

    1. 对等协商双方都支持的秘钥算法；
    2. 基于非对称秘钥的信息传输加密和身份认证，基于KPI证书的身份认证；
    3. 基于对称密钥的数据传输保密；

建立SSL连接
SSL握手过程

–

  1. 客户端向服务器端发送”ClientHello”消息，内容包括：

    • 客户端支持的TLS协议版本(Version: TLS 1.2);
    • 一个客户端生成的随机数(Random)，该随机数将用于生成”会话密钥”;
    • 客户端所支持的加密算法列表(Cipher Suites);
    • 客户端所支持的压缩算法(Compression Methods);

  2. 服务器端向客户端发送”ServerHello”消息，内容包括：

    • 确认使用TLS协议的版本(Version: TLS 1.2);
    • 一个服务器生成的随机数(Random)，该随机数将用于生成”会话密钥”;
    • 确认使用加密算法(Cipher Suite: TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256)；

  3. 服务器端发送证书给客户端；

  4. 客户端收到服务器证书后，需要验证证书是否可信(证书是否为可信任机构颁发、证书中的域名是否与预期域名一致、证书是否过期)，如果证书没有问题，客户端会从证书内获得服务器公钥，然后向服务器发送消息，消息内容包括：

    • 一个客户端随机数，该随机数被服务器公钥加密，以防止被窃听；
    • 编码改变通知(Change Cipher Spec)，表示随后的消息将用双方协商的加密方法和密钥发送；
    • 客户端握手结束通知，表示客户端的握手阶段已经结束；

  5. 服务器端收到客户端的随机数后，计算并生产本次会话所使用的”会话密钥”，然后向客户端发送消息，内容包括：

    • 编码改变通知，表示随后的信息都将用双方商定的加密方法和密钥发送；
    • 服务器握手结束通知，表示服务器的握手阶段已经结束；

TCP滑动窗口

发表于 2017-05-11

滑动窗口(Sliding Window)

  • 滑动窗口是什么？

    • 滑动窗口是一种技术(或者一种设计)，它被用来做网络流量控制的。

  • 滑动窗口在TCP协议中的实现方式

    • 在TCP协议头中有一个字段Window Size，该字段是接收端将自己缓冲区还可以接收数据大小通知发送端；发送端根据该数值来判断接收端的处理能力，并调整发送数据包大小，使接收端有足够的能力来处理接收到的数据包；

发送端滑动窗口

滑动前示意图

  • Category #1标识发送端已经发送出去的并且已经被接收端确认(收到接收端Ack)的数据;
  • Category #2标识发送端已经发送出去的但是没有被接收端确认(没有收到接收端Ack)的数据;
  • Category #3标识发送端没有发送出去的并且能够被接收端接收的数据;
  • Category #4标识发送端没有发送出去的而且接收端没有缓冲空间接收的数据(也就是说该部分数据不能够被接收端接收处理);

  • Category #2标识发送端已经发送出去的但是没有被接收端确认(没有收到接收端Ack)的数据;
  • Category #3标识发送端没有发送出去的而且接收端没有缓冲空间接收的数据(也就是说该部分数据不能够被接收端接收);

发送端与接收端滑动窗口变化示意图

过程分析

  1. TCP连接建立后发送端发送窗口与接收端的接收窗口大小均为360 Bytes，此时可发送数据大小为360 Bytes;
  2. 发送端发出140 Bytes到接收端，此时可发送数据大小变为220 Bytes，但发送窗口没有改变;
  3. 接收端收到140 Bytes的数据包后接收窗口大小变为260，同时发送Ack给发送端，通知发送端自己还能接收并处理260 Bytes的数据；
  4. 发送端收到Ack数据包后，根据Window Size的数值来改变自己的发送窗口大小(变为260 Bytes);
  5. 经过三次数据传递，最终发送端窗口大小变为0，也就是发送端不能再发送数据包

如果接收端太过繁忙不能及时处理接收窗口数据的时候就会出现发送端窗口被关闭(大小变为0)的情况，这时客户端需要每隔一个时间间隔向接收端发送一个查询数据包，来询问接收端的接收窗口大小，接收端通过Ack数据包将Window的大小通知发送端，经过几次(一般为3次)询问后，如果接收端窗口依然为0，TCP一般会发出RST数据包将连接断开。这种技术被称为Zero Window Probe(ZWP)。

Name Description Length

Destination Port 目标端口 2 bytes - 16 bits
Sequence Number 序列号 4 bytes - 32 bits
Acknowledgment Number 确认号 4 bytes - 32 bits
Header Length 协议头长度 4 bits
Flags 标志 12 bits
Window Size Value 窗口尺寸 2 bytes - 16 bits
CheckSum 校验和，该校验和覆盖整个TCP报文(包括首部和数据)，由发送端生成，由接收端验证 2 bytes - 16 bits
Urgent Pointer 紧急数据指针 2 bytes - 16 bits
Options 选项 最大40 bytes
  • TCP基本协议头(不包含Options)占用20个字节

TCP标志位

Name Description Length
Reserved – 1 bit
nonce – 1 bit
CWR 流量控制 1 bit
ECN 拥塞窗口 1 bit
URG 紧急指针有效性 1 bit
ACK 确认序号有效性 1 bit
PSH 接收方应尽快将当前报文段交付给应用层处理 1 bit
RST 重新建立连接 1 bit
SYN 同步序号，在发起一个TCP连接时使用 1 bit
FIN 发送端完成发送任务 1 bit

  • CWR(Congestion Window Reduced)与ECN(ECN-Echo, Explicit Congestion Notification)CWR标志与ECN标志是被共同使用的一对标志位，ECN是由接收端发出，目的是通知发送端:报文出现了拥塞，需要发送端控制发包速度；CWR是由发送端发出，是对接收端ECN报文的应答，目的是通知接收端:已经降低了拥塞窗口；在TCP会话建立的时候会协商发送端与接收端是否支持流控，如果发送端支持，则在发送SYN包的时候将CWR与ECN标志置为1；如果接收端也支持，则在发送ACK包的时候将ECN标志置为1；

  • URG(紧急数据标志位) 当发送端发送的报文需要被接收端优先处理，则需要设置该标志位；该标志位需要与Urgent Pointer(紧急指针)配合使用(紧急指针会设置相应的数据偏移)；接收端收到URG报文后，不会将该报文缓存在接收窗口，而是直接交给应用层处理；

  • PSH该标志为主要用于通知接收端将缓存中的报文与当前报文尽快交付给应用层处理；

  • SYN(Synchronize)同步标志位，在新建一个TCP连接的时候，发送端与接收端都需要设置该标志位;

  • ACK(Acknowledgement)应答标志位，确认收到数据标志位；

  • FIN(Finish)结束标志位

  • RST(Reset)重连标志位

Sequence Number 与 Acknowledgment Number计算方式

  • ISN：在TCP连接建立的时候，发送端与接收端都会按照相应的算法生成一个对于该TCP连接唯一性的初始化的序列号(Initial Sequence Number)，该序列号被称为ISN；

  • TCP连接建立的时候(三次握手)：前两次握手的时候发送端与接收端的Seq由各自系统按照相应的算法产生，最后一次握手时发送端的Seq = ISN + 1，接收端的Ack = 发送端Seq + 1;

  • TCP连接建立完成后：接收端的Seq = 发送端Ack，接收端的Ack = 发送端Seq + 发送端数据长度Len;

TCP Options

Options Kind说明

Kind Description
0 EOL选项结束标志
1 NOP无操作类型(用于字节填充)
2 MSS最大报文长度
3 WS窗口扩大因子
4-7 SACK相关
8 TS时间戳

Options字段详解

  • MSS(Maximum Segment Size)

    • Kind: Maximum Segment Size (2)
    • Length: 4

    • MSS Value: 1360

      MSS是指一个TCP连接中的最大报文传输长度，在三次握手的时候，由SYN报文确定该数值，该值是由网络设备属性得出。MSS很常见的值是1460(MTU1500 - IP头部长度 - TCP头部长度)

  • SACK(TCP SACK Permitted Option)
    • Kind: SACK Permitted (4)
    • Length: 2
_SACK被称为选择性确认(Selective Acknowledgment), 在TCP通信时，如果发送序列中某个数据包丢失，TCP会通过重传最后确认的包之后的数据包，造成的结果就是可能重新发送已经正确传输过的数据包，这样会降低TCP的性能。为避免上述问题，产生了SACK技术，即：只重新发送丢失的数据包，而不用发送最后确认包之后的所有数据包；接收端会通过相应的机制通知发送端哪些数据包丢失，哪些数据被重发，哪些数据已经收到_

_在TCP握手阶段，连接双方在各自的SYN包中标明己方是否支持SACK_

_在通信阶段，SACK包通常是由接收端产生；如果TCP接收端收到非预期序列号的数据包时，如果该序列号小于预期序列号，则表明该数据包是通过网络复制或是发送端重发产生的，该数据包是可以丢弃的；如果收到的数据包序列号大于预期的序列号，则表明有数据包被丢弃或延迟，此时接收端发送SACK数据包通知发送端出现了丢包_

_TCP发送端维护着一个重发数据队列，该队列中的数据包都是没有被接收端确认正确接收，数据包在没有被确认时是不能释放的，该队列的数据包都有一个SACKED标志位，该标志位标明该数据包是否被SACK过，对于已经被标识过数据包，在重新发送时将被跳过_
  • WS(Window Scale)
    • Kind: Window Scale (3)
    • Length: 3
    • Shift count: 5
_虽然在TCP头部已经包含了Window Size Value信息，但是该字段是一个16 bits的值，用该值来表示窗口大小不能满足现代网络的需求，所以TCP通过WS选项来扩展窗口大小_

_WS的取值范围是0~14，表示Window Size的数值向左移动的位数，这样实际的窗口大小可以扩展到31 bits_
  • Timestamps
    • Kind: Time Stamp Option (8)
    • Length: 10
    • Timestamp Value: 456278103
    • Timestamp echo reply: 0
_Timestamps包含两个32 bits的值：**TSval**(Timestamp Value)与**TSecr**(Timestamp echo reply)_

_发送端发包时，将当前时间写入**TSval**，接收端回应时，将发送端的TSval写入**TSecr**即可_

_TS选项有两个作用：**1.**辅助精确计算RTT，接收端只需要在回应报文里用当前时间减去TSecr即可；**2.**防止Sequence回绕_
